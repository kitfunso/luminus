# Integrated Intelligence Rail Design

Date: 2026-03-24
Product: Luminus web dashboard
Status: Approved design direction
Chosen approach: Integrated intelligence rail

## Summary

Luminus should move from a build-time snapshot dashboard with multiple competing floating panels to a live runtime dashboard with one coordinated intelligence surface.

This design does three things together:

1. Replaces build-time-only dashboard data with a server-backed live data layer that auto-refreshes and supports manual refresh.
2. Resolves the current layout drift by anchoring the day-ahead market card to the true bottom-left corner and consolidating Forecast vs Actual plus Outage Radar into one right-side intelligence rail.
3. Replaces the current modal onboarding with a guided, replayable walkthrough that opens real layers and panels, but lets users pause, skip, revisit, and continue exploring at their own pace.

## Problem Statement

The current dashboard has three structural issues:

1. Data shown in the dashboard is loaded from bundled `/data/*.json` files generated at build time, so the product does not behave like a live market dashboard.
2. The bottom-left market card is offset inward by a hardcoded desktop `left` value instead of being anchored to the viewport corner.
3. Forecast vs Actual and Outage Radar behave like separate floating panels, which creates overlap, inconsistent focus, and too many competing reading zones.

The existing onboarding is also too light. It explains the product in abstract text, but it does not teach the actual interaction model.

## Goals

- Deliver live runtime dashboard data backed by Next server endpoints.
- Support both auto-refresh and user-triggered manual refresh.
- Preserve the last successful payload during partial provider failures.
- Make freshness and fallback state visible to the user.
- Anchor the day-ahead card to the real bottom-left corner on desktop.
- Consolidate map intelligence surfaces into one right-side rail.
- Create a guided first-run tutorial that manipulates the real UI.
- Let users skip, pause, resume, and replay the tutorial.
- Preserve mobile usability with the same information architecture.

## Non-Goals

- Redesign the brand or overall visual identity from scratch.
- Turn the first viewport into a promotional landing page.
- Remove existing data-fetch scripts used for bootstrap or fallback.
- Build a fully locked onboarding mode that prevents free exploration.
- Re-architect the entire state layer beyond what is needed for live data, rail coordination, and tutorial control.

## User Experience Principles

- One map, one control plane, one intelligence rail.
- Live data must look live and expose trust cues.
- Tutorial guidance should reveal real product behavior, not replace it.
- Advanced users should never be trapped in onboarding.
- Motion should communicate changes in focus, freshness, and context, not decorate the screen.

## Current-State Findings

- Client data loading is currently handled by `web/src/lib/data-fetcher.ts` and reads bundled JSON from `/data/*.json`.
- Bundled JSON is generated by `web/scripts/fetch-data.js` during `prebuild`.
- The bottom-left legend/card offset is controlled by `.map-legend` in `web/src/app/globals.css`, which currently uses a fixed desktop `left: 304px`.
- `web/src/components/Map.tsx` conditionally mounts separate `OutageRadar` and `ForecastPanel` surfaces rather than routing both through one container.
- `web/src/components/Onboarding.tsx` is a simple modal with static copy and no UI orchestration.

## Proposed Architecture

### 1. Live Runtime Data Layer

Add server-backed data endpoints inside the Next app for the main dashboard datasets:

- `prices`
- `flows`
- `outages`
- `forecasts`
- `history`

These endpoints should:

- Fetch fresh data from the existing provider logic or shared server-side adapters.
- Normalize payload shape for the client.
- Return freshness metadata alongside the payload.
- Support cached revalidation on the server to avoid provider abuse.
- Fall back to the latest successful bootstrap/static dataset when live fetch is unavailable.

### 2. Shared Client Live-Data Controller

Create a shared client-side live-data controller used by the map and right-side rail.

Responsibilities:

- Initial load from runtime endpoints.
- Auto-refresh on an interval.
- Manual refresh on demand.
- Per-dataset status handling.
- Last-successful-payload retention.
- UI-friendly metadata formatting.

The client state model should include, per dataset:

- `data`
- `lastUpdated`
- `source`
- `isLoading`
- `isRefreshing`
- `isStale`
- `hasFallback`
- `error`

The controller should expose a unified refresh action and allow selective refresh when needed.

### 3. Right-Side Intelligence Rail

Replace separate floating `OutageRadar` and `ForecastPanel` panels with one stable right-side intelligence rail.

The rail becomes the single contextual reading zone for:

- Morning Brief
- Outage Radar
- Forecast vs Actual
- Pipeline reads
- Future intelligence modules

The rail may support internal modes or tabs, but it should remain one physical container with one width and one placement on desktop.

### 4. Anchored Bottom-Left Market Card

The bottom-left market card should be anchored to the viewport margin rather than offset to compensate for the sidebar by default.

Desktop behavior:

- sits in the true lower-left corner with consistent margin
- remains stable while map state changes
- only shifts if the sidebar physically overlaps it

Mobile behavior:

- continues to respect the mobile sheet/action-bar system
- keeps a readable lower-left position without clashing with touch controls

### 5. Guided Tutorial System

Replace the current modal-only onboarding with a guided tutorial system that can drive the real interface.

The tutorial must:

- start from a compact welcome state
- offer `Start tour` and `Skip for now`
- open relevant tabs, layers, and rail modes
- spotlight actual UI elements
- support `Next`, `Back`, `Skip`, and `Pause and explore`
- persist progress locally
- leave behind a resumable checklist when paused
- remain replayable from the product UI

## Detailed Design

### Live Data UX

Add a lightweight freshness strip or live status region near the top of the intelligence surface.

It should show:

- live/snapshot state
- updated timestamp
- auto-refresh cadence
- manual refresh action

Example states:

- `Live | Updated 09:42 UTC | Auto-refresh 2m`
- `Refreshing...`
- `Stale | Last good update 09:42 UTC`
- `Live unavailable | showing last successful snapshot`

Important rule:

Never silently present fallback or synthetic data as if it were fresh live data.

### Intelligence Rail UX

The right-side rail should behave as a single coordinated surface.

Desktop:

- fixed right position
- stable width
- shared enter/exit motion
- internal section switching for Outages, Forecast, Morning Brief, and related reads

Mobile:

- same information model
- rendered as a bottom sheet
- tutorial targets sheet controls and section changes cleanly

Interaction model:

- toggling `Outage Radar` opens the rail in outage mode
- toggling `Forecast vs Actual` opens the rail in forecast mode
- opening the brief or dashboard summary uses the same rail container
- closing the rail closes the active intelligence mode rather than removing arbitrary layers

### Bottom-Left Card UX

The current day-ahead card should feel like a permanent orientation aid.

Changes:

- anchor it to the real bottom-left corner
- preserve a clear margin from the viewport edge
- keep it visually subordinate to the map, but stable
- avoid sidebar-compensation offsets unless overlap is real

The component should also become a better trust surface by exposing:

- current market label
- freshness state when relevant
- maybe a compact status dot if this becomes the preferred always-on live badge

### Tutorial UX

The tutorial combines guided progression with self-paced exploration.

#### Entry

On first run:

- show a compact welcome surface, not a blocking full-screen modal
- present:
  - `Start tour`
  - `Skip for now`

If skipped:

- persist dismissal
- keep `Replay tour` available in the app

#### Guided Steps

Recommended step sequence:

1. Price heatmap and live status
2. Country click interaction
3. Cross-border flows
4. Outage Radar in the intelligence rail
5. Forecast vs Actual in the intelligence rail
6. Morning Brief / summary mode
7. Filters, presets, and revisit controls

Each step should:

- highlight a real UI target
- contain short copy
- provide one clear action
- support next/back/skip

#### Pause and Explore

At any step, the user can choose `Pause and explore`.

When paused:

- the spotlight overlay closes
- a small progress chip or checklist remains available
- the current step stays resumable

The residual checklist acts as a non-blocking guide for users who want both structure and freedom.

#### Replay

Add `Replay tour` to a durable location, likely the overview tab or a help/tutorial section in the sidebar.

## Motion Design

Ship a few intentional motions:

- rail mode transitions slide/fade instead of stacking panels
- tutorial spotlight softly expands and settles around the target
- refresh state uses subtle pulse/spinner behavior
- completion or resumption of a tutorial step feels deliberate, not abrupt

Avoid generic animation spam. Motion should explain where attention moved and why.

## Error Handling

### Provider Failure

If a live provider fails:

- keep the last successful payload visible
- mark the affected section stale
- expose the failure state in freshness copy
- do not collapse the entire dashboard if one dataset fails

### Mixed Freshness

If datasets are out of sync:

- show freshness at the section level where necessary
- allow prices to be fresh while outages or forecasts are stale

### First Load Failure

If live runtime fetch fails on first load:

- hydrate from bundled bootstrap data if available
- label it as snapshot/fallback
- allow manual retry

## Accessibility

- Spotlight targets must remain keyboard reachable.
- Tutorial controls must be fully keyboard operable.
- The tutorial should not trap focus unless intentionally acting as the active dialog surface.
- Status messaging for refresh/fallback should be announced appropriately for assistive technology.
- Motion should respect reduced-motion preferences.

## Testing Strategy

### Unit / State Tests

Cover live-data state transitions:

- first load
- refresh success
- refresh failure with last-good retention
- stale timeout handling
- fallback labeling
- partial dataset failure

### Component Tests

Cover:

- intelligence rail mode switching
- rail close behavior
- tutorial progression
- pause/resume/skip/replay
- bottom-left card anchoring logic

### Responsive Testing

Verify:

- desktop lower-left anchoring
- no collision between rail and anchored market card
- mobile bottom-sheet rail behavior
- tutorial overlays target correct elements on both desktop and mobile

### Manual QA

Run checks for:

- slow network
- provider outage
- stale timestamp rendering
- manual refresh feedback
- repeat visit after tutorial dismissal
- replaying the tour after completing it

## Implementation Notes

- Preserve the current visual language and avoid introducing card-heavy clutter.
- The first viewport must still read as one cohesive dashboard composition.
- Follow the current mobile sheet behavior where appropriate rather than inventing a separate mobile architecture.
- The existing build-time fetch scripts should remain useful as bootstrap/fallback inputs even after runtime data is added.

## Acceptance Criteria

- Dashboard data is fetched from live server-backed endpoints at runtime.
- Auto-refresh and manual refresh both work.
- Freshness and fallback state are visible in the UI.
- The bottom-left market card is anchored to the real corner on desktop.
- Forecast and outage content no longer overlap as separate floating panels.
- One right-side rail handles intelligence modes consistently.
- New users can run a guided tour that opens real layers and panels.
- Users can skip, pause, resume, and replay the tutorial.
- Mobile behavior remains coherent and usable.

## Open Questions

None blocking for the implementation plan. The chosen direction is clear enough to proceed.
